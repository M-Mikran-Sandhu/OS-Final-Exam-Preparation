Chapter 1: Introduction to the Course

1.1 What is an Operating System?

An Operating System (OS) is a software that acts as an intermediary between the user and the computer hardware. It manages all the hardware resources and provides services to application programs. The OS controls the execution of all kinds of programs and makes the computing system convenient to use.

Key Functions:

  • Resource management (CPU, memory, I/O devices)
  • Process and thread management
  • Memory management
  • File system management
  • Device management
  • Security and protection

1.2 Why do we need an OS?

Operating systems are essential for several reasons:

  • Hardware Abstraction: Provides a uniform interface to diverse hardware
  • Resource Allocation: Efficiently distributes CPU, memory, and I/O resources
  • Process Management: Manages multiple processes running simultaneously
  • Security: Protects data and resources from unauthorized access
  • User Interface: Provides convenient way to interact with computer
  • File Management: Organizes and manages data storage

1.3 System Boot

System boot is the process of starting up a computer and loading the operating system into memory. The boot process involves:

  1. Power-on: CPU executes firmware code from ROM
  2. POST (Power-On Self Test): Hardware diagnostics run
  3. Bootloader: Loads the OS kernel into memory
  4. Kernel Initialization: OS initializes system resources
  5. System Services: Background services and daemons start
  6. User Login: System ready for user interaction

1.4 Computer-System Architecture

Modern computer systems consist of several key components:

  • CPU (Central Processing Unit): Executes instructions
  • Memory: RAM for temporary storage, ROM for firmware
  • I/O Devices: Input/output controllers and devices
  • System Bus: Connects all components
  • Storage: Permanent data storage (HDD, SSD)

1.5 Operating System Concepts

Multiprogramming: Multiple programs loaded in memory simultaneously, OS switches between them

Multitasking: CPU rapidly switches between tasks, giving appearance of parallel execution

Multiprocessing: Multiple CPUs work simultaneously on different tasks

1.6 Interrupts

Hardware Interrupts: Generated by hardware devices (I/O, timer)

Software Interrupts: Generated by executing software instructions (system calls)

Exceptions: Unusual conditions during program execution (division by zero, page fault)

1.7 Interrupt Service Mechanism

When an interrupt occurs:

  1. CPU stops current execution
  2. Saves current state (registers, program counter)
  3. Identifies interrupt source
  4. Executes Interrupt Service Routine (ISR)
  5. Restores saved state and resumes execution

1.8 Protection Mechanisms

Memory Protection: Prevents one process from accessing another's memory

I/O Protection: Only OS can directly access I/O devices

CPU Protection: Timer interrupts prevent process monopolizing CPU

1.9 Dual-mode of Operation

User Mode: Restricted execution mode for applications

Kernel Mode: Privileged mode for OS with full hardware access

Mode switching occurs via system calls and interrupts.

Chapter 2: Operating System Services

2.1 User Interface

The OS provides interfaces for users to interact with the computer system.

Command Line Interface (CLI)

Users type commands to interact with the OS. Examples: bash, cmd.exe, PowerShell

  • Direct control over system
  • Powerful for automation
  • Steep learning curve

Graphical User Interface (GUI)

Visual interface with windows, icons, and menus. Examples: Windows Desktop, macOS, GNOME

  • User-friendly and intuitive
  • Visual representation of files and programs
  • Point-and-click interaction

2.2 Programmer Interface

API (Application Programming Interface): Set of functions and protocols for building applications

System Calls: Interface between user programs and OS kernel

System calls provide access to OS services like file I/O, process creation, and memory allocation.

2.3 Operating System Design

OS design involves deciding how to structure the system for efficiency and maintainability.

Simple Structure

No clear structure, monolithic kernel. Example: Early MS-DOS

  • Limited functionality
  • Poor isolation between components
  • Difficult to maintain

Monolithic Kernel

Entire OS runs in kernel mode as a single program

  • Fast performance
  • Difficult to debug and maintain
  • One bug can crash entire system
  • Examples: Linux, Unix

Microkernel

Only essential services in kernel; other services run as user-level processes

  • Better modularity and reliability
  • Easier to extend and maintain
  • More context switches (performance overhead)
  • Examples: QNX, MINIX

Layered Architecture

OS organized in layers, each using services from lower layers

  • Clear separation of concerns
  • Easier to understand and debug
  • Performance overhead from layer crossing

Modular Structure

Core kernel with loadable modules for additional functionality

  • Flexibility to add/remove features
  • Better than monolithic for maintenance
  • Examples: Modern Linux kernels

Hybrid Systems

Combination of microkernel and monolithic approaches

  • Mac OS X: XNU kernel (hybrid of Mach microkernel and BSD monolithic)
  • iOS: Based on Mac OS X architecture
  • Android: Linux kernel with custom framework

Chapter 3: Process Management

3.1 Process States

A process goes through several states during its lifetime:

  • New: Process is being created
  • Running: Process is executing on CPU
  • Waiting: Process waiting for I/O or event
  • Ready: Process ready to run, waiting for CPU
  • Terminated: Process has finished execution

3.2 Process Control Block (PCB)

Data structure containing information about a process:

  • Process ID (PID)
  • Program counter
  • CPU registers
  • Memory management information
  • I/O status information
  • Accounting information

3.3 Process Scheduling

Determining which process gets CPU time.

Scheduling Queues

  • Job Queue: All processes in system
  • Ready Queue: Processes ready to execute
  • Device Queue: Processes waiting for I/O

Schedulers

  • Long-term Scheduler: Selects processes for ready queue (job scheduler)
  • Medium-term Scheduler: Swaps processes between memory and disk
  • Short-term Scheduler: Selects process from ready queue for CPU (dispatcher)

3.4 Context Switching

Process of saving state of current process and loading state of next process

  • Occurs when process yields CPU
  • Involves saving/restoring PCB
  • Pure overhead - no useful work done

3.5 UNIX Programming

Child Process Creation: Creating new processes from existing ones

Fork(): System call that creates a new process (child) identical to parent

  • Returns 0 to child, child PID to parent
  • Both processes continue from fork() call
  • Child inherits parent's memory space

Process Invocation with Exec(): System call that replaces process image with new program

  • Loads new program into memory
  • Replaces current process image
  • Does not create new process

3.6 Orphan and Zombie Processes

Orphan Process: Child process whose parent has terminated

Zombie Process: Child process that has terminated but parent hasn't collected its exit status

Chapter 4: Interprocess Communication (IPC)

4.1 IPC Methods

Mechanisms for processes to communicate and synchronize:

Anonymous Pipes

One-way communication channel between related processes

  • Created with pipe() system call
  • Unidirectional data flow
  • Used for parent-child communication
  • Data lost when pipe is closed

Shared Memory

Multiple processes access common memory region

  • Fastest IPC method
  • Requires synchronization mechanisms
  • Processes must coordinate access
  • System calls: shmget(), shmat(), shmdt()

Other IPC Methods

  • Named Pipes (FIFOs): Bidirectional communication
  • Message Queues: Asynchronous message passing
  • Sockets: Network communication
  • Signals: Asynchronous notifications

4.2 IPC Examples

Practical examples of IPC implementation in UNIX systems

  • Producer-consumer problem using pipes
  • Shared memory with semaphores
  • Message passing between processes

Chapter 5: CPU Scheduling

5.1 Basic Concepts

CPU scheduling is the process of determining which process gets CPU time.

5.2 CPU I/O Burst Cycle

Process alternates between CPU execution and I/O operations

  • CPU Burst: Time process uses CPU
  • I/O Burst: Time process waits for I/O
  • Scheduler must manage these bursts efficiently

5.3 Preemptive vs Non-preemptive Scheduling

Non-preemptive: Process runs until it yields CPU voluntarily

Preemptive: OS can interrupt process and assign CPU to another

5.4 Scheduling Criteria

  • CPU Utilization: Percentage of time CPU is busy
  • Throughput: Number of processes completed per unit time
  • Turnaround Time: Time from submission to completion
  • Waiting Time: Total time process spends in ready queue
  • Response Time: Time from submission to first response

5.5 Scheduling Algorithms

First-Come, First-Served (FCFS)

Processes executed in order they arrive

  • Simple to implement
  • Non-preemptive
  • Convoy effect (short jobs wait for long ones)

Shortest-Job-First (SJF)

Process with shortest CPU burst executes first

  • Minimizes average waiting time
  • Difficult to predict burst time
  • Can starve long processes

Priority Scheduling

Each process assigned priority; higher priority executes first

  • Flexible and widely used
  • Risk of starvation for low-priority processes
  • Solution: aging (increase priority over time)

Round-Robin Scheduling

Each process gets fixed time slice (quantum)

  • Fair allocation of CPU time
  • Preemptive
  • Performance depends on quantum size

Multilevel Queue Scheduling

Ready queue divided into separate queues for different process types

  • Each queue has own scheduling algorithm
  • Fixed priority between queues

Multilevel Feedback Queue Scheduling

Processes can move between queues based on behavior

  • Most flexible scheduling algorithm
  • Adapts to process characteristics
  • Complex to implement

Chapter 6: Threads

6.1 What is a Thread?

A thread is the smallest unit of execution within a process. It's a lightweight process that shares memory with other threads in the same process.

6.2 Single-threaded vs Multithreaded Processes

Single-threaded: One thread of execution per process

Multithreaded: Multiple threads within same process

  • Threads share memory space
  • Faster context switching between threads
  • Easier communication between threads
  • Requires synchronization

6.3 Parallelism

Data Parallelism: Same operation on different data elements

Task Parallelism: Different operations on same or different data

6.4 Multithreading Models

Many-to-One Model

Many user-level threads mapped to one kernel thread

  • Efficient context switching
  • One blocking thread blocks all threads
  • Cannot use multiprocessor

One-to-One Model

Each user thread mapped to kernel thread

  • Better concurrency
  • More overhead
  • Can use multiprocessor
  • Examples: Windows, Linux

Many-to-Many Model

Many user threads mapped to many kernel threads

  • Best of both worlds
  • More complex to implement
  • Examples: Solaris, HP-UX

6.5 UNIX Threading

Thread Libraries: POSIX Threads (Pthreads)

Pthreads Creation

pthread_create() creates new thread

  • Specifies thread function
  • Passes arguments to thread

Parameter Passing to Threads

Arguments passed via void pointer

Returning Values from Threads

pthread_join() waits for thread completion and retrieves return value

6.6 Threading Issues

Thread Cancellation: Terminating thread before completion

Implicit Threading: Compiler/runtime creates threads

Thread Pools: Pre-created threads waiting for work

Chapter 7: Process Synchronization

7.1 The Critical Section Problem

Multiple processes accessing shared resources can lead to inconsistent data. The critical section is code that accesses shared resources.

Requirements for solution:

  • Mutual Exclusion: Only one process in critical section
  • Progress: Process not in critical section cannot block others
  • Bounded Waiting: Process waits finite time to enter

7.2 Solutions to Critical Section Problem

Software Solutions

Peterson's Solution: Uses flags and turn variable

  • Guarantees mutual exclusion
  • Works for two processes
  • Busy waiting (inefficient)

Hardware Solutions

Disabling Interrupts: Disable interrupts in critical section

  • Simple but inefficient
  • Not suitable for multiprocessor

Atomic Instructions: Test-and-set, compare-and-swap

  • Hardware-supported indivisible operations
  • Efficient and reliable

7.3 Synchronization Primitives

Mutex (Mutual Exclusion Lock)

Binary lock - locked or unlocked

  • lock() - acquire mutex
  • unlock() - release mutex
  • Prevents multiple threads from accessing resource

Semaphore

Integer variable with two atomic operations

  • wait() (P): Decrement semaphore, wait if negative
  • signal() (V): Increment semaphore
  • Binary Semaphore: Values 0 and 1 (like mutex)
  • Counting Semaphore: Any non-negative value

7.4 Deadlock and Starvation

Deadlock: Two or more processes waiting indefinitely for each other

Starvation: Process never gets CPU time

Chapter 8: Memory Management

8.1 What is Memory Management?

Simple Analogy: Think of computer memory like a library. The OS is the librarian who decides where to place books (programs) and how to organize them so they can be quickly found and retrieved.

Memory management is the process of managing the computer's main memory (RAM). It allocates memory to processes when they need it and deallocates it when they're done.

8.2 Paging - The Modern Approach

What is Paging? Paging divides both logical memory (what programs see) and physical memory (actual RAM) into fixed-size chunks called pages and frames.

Key Terms Explained Simply:

  • Page: A fixed-size block of logical memory (typically 4KB). Think of it as a page in a book.
  • Frame: A fixed-size block of physical memory matching page size. Think of it as a shelf in the library.
  • Logical Memory: The memory address space that a program sees (0 to max address). Like the book's page numbers.
  • Physical Memory: The actual RAM addresses where data is stored. Like the actual shelves in the library.
  • Page Table: A mapping table that translates logical page numbers to physical frame numbers. Like a library catalog.
  • TLB (Translation Lookaside Buffer): A fast cache that stores recent page-to-frame mappings. Like remembering where you just found a book.

8.3 How Paging Works - Step by Step

Example: A program needs to access memory address 4096 (in logical memory)

  1. CPU divides address into: Page Number (1) and Offset (0)
  2. Check TLB for quick lookup - if found, use it
  3. If not in TLB, consult Page Table
  4. Page Table says: Page 1 is in Frame 5
  5. Calculate physical address: (Frame 5 × 4096) + Offset = 20480
  6. Access data at physical address 20480
  7. Add mapping to TLB for next time

8.4 Internal vs External Fragmentation

Internal Fragmentation: Wasted space INSIDE allocated memory

  • Example: A program needs 5KB but gets allocated one 4KB page + one 4KB page = 8KB total. 3KB is wasted inside the allocated space.
  • Cause: Fixed page size doesn't match exact program needs
  • Paging Impact: Minimal because pages are small (usually 4KB)

External Fragmentation: Wasted space BETWEEN allocated memory blocks

  • Example: Memory layout: [Program A: 10KB] [Free: 2KB] [Program B: 10KB] [Free: 3KB]. The 2KB and 3KB gaps can't be used if a program needs 5KB.
  • Cause: Programs are loaded and removed, leaving scattered free spaces
  • Paging Impact: Paging ELIMINATES external fragmentation because pages are fixed-size and can go anywhere

8.5 Advantages of Paging

  • Eliminates external fragmentation
  • Simple to implement and manage
  • Allows programs larger than physical memory (with virtual memory)
  • Easy memory allocation and deallocation

8.6 Paging Structure Diagram

Visual Representation:

Chapter 9: Deadlock

9.1 What is Deadlock?

Simple Analogy: Imagine two people in a narrow hallway. Person A is holding a book and wants a pen (which Person B has). Person B is holding a pen and wants the book (which Person A has). Neither can move forward - they're stuck. This is deadlock!

A deadlock occurs when two or more processes are blocked forever, waiting for each other to release resources they need.

9.2 Necessary Conditions for Deadlock

All four conditions must be present simultaneously for deadlock to occur:

  • 1. Mutual Exclusion: Resource can only be used by one process at a time. (Like a book - only one person can read it)
  • 2. Hold and Wait: Process holds resources while waiting for others. (Person A holds the book while waiting for the pen)
  • 3. No Preemption: Resources cannot be forcibly taken away. (You can't just grab the book from Person A)
  • 4. Circular Wait: Processes form a cycle waiting for each other. (A waits for B, B waits for A)

9.3 Deadlock Example - Two Processes

Process A:

  1. Acquires Resource 1 (lock on file A)
  2. Tries to acquire Resource 2 (lock on file B) → BLOCKED

Process B:

  1. Acquires Resource 2 (lock on file B)
  2. Tries to acquire Resource 1 (lock on file A) → BLOCKED

Result: Both processes are blocked forever! Neither can proceed.

9.4 Deadlock Handling Strategies

1. Deadlock Prevention: Ensure one of the four conditions cannot occur

  • Break Circular Wait: Assign resources in order (always request Resource 1 before Resource 2)
  • Allow Preemption: OS can take back resources if needed

2. Deadlock Avoidance: Detect unsafe states before they occur

  • Banker's Algorithm: Before allocating resource, check if system remains safe
  • Safe State: A state where all processes can complete without deadlock

3. Deadlock Detection and Recovery: Allow deadlock to occur, then detect and recover

  • Detect: Check for cycles in resource allocation graph
  • Recover: Terminate processes or preempt resources

4. Ignore Deadlock: (Most common in modern OS)

  • Assume deadlock is rare
  • Let user restart hung processes
  • Used by Windows, Linux, macOS

9.5 Resource Allocation Graph (RAG)

What is RAG? A directed graph showing which processes hold which resources and which processes are waiting for resources.

  • Nodes: Processes (circles) and Resources (squares)
  • Edges: Request edges (process → resource) or Allocation edges (resource → process)
  • Cycle Detection: If there's a cycle, deadlock exists

Chapter 10: The Dining Philosophers Problem

10.1 Problem Definition

The Scenario: Five philosophers sit around a circular table. Between each pair of philosophers is one chopstick (5 chopsticks total). Each philosopher alternates between thinking and eating. To eat, a philosopher needs BOTH chopsticks (left and right).

The Challenge: Design a synchronization solution so that philosophers can eat without deadlock or starvation.

10.2 Why This is a Problem

Deadlock Scenario:

  1. Philosopher 1 picks up left chopstick
  2. Philosopher 2 picks up left chopstick
  3. Philosopher 3 picks up left chopstick
  4. Philosopher 4 picks up left chopstick
  5. Philosopher 5 picks up left chopstick
  6. Now each philosopher tries to pick up their right chopstick → ALL BLOCKED!

All philosophers are waiting for a chopstick held by their neighbor. Nobody can eat. Nobody releases their chopstick. DEADLOCK!

Starvation Scenario: Even without deadlock, some philosophers might never get to eat if others keep hogging the chopsticks.

10.3 Solution 1: Semaphore-Based Solution

Approach: Use semaphores to control access to chopsticks

Semaphore chopstick[5] = {1, 1, 1, 1, 1}; // All available

Philosopher i:
  while(true) {
    think();
    wait(chopstick[i]);           // Pick up left
    wait(chopstick[(i+1)%5]);     // Pick up right
    eat();
    signal(chopstick[i]);         // Put down left
    signal(chopstick[(i+1)%5]);   // Put down right
  }

Problem: This still causes deadlock! If all philosophers pick up their left chopstick simultaneously, all are blocked.

10.4 Solution 2: Asymmetric Solution (Prevents Deadlock)

Approach: Make one philosopher pick up chopsticks in different order

Philosopher i (i != 4):
  while(true) {
    think();
    wait(chopstick[i]);           // Pick up left first
    wait(chopstick[(i+1)%5]);     // Then pick up right
    eat();
    signal(chopstick[i]);
    signal(chopstick[(i+1)%5]);
  }

Philosopher 4 (different order):
  while(true) {
    think();
    wait(chopstick[0]);           // Pick up right first!
    wait(chopstick[4]);           // Then pick up left
    eat();
    signal(chopstick[0]);
    signal(chopstick[4]);
  }

Why it works: Breaking the circular wait condition prevents deadlock!

10.5 Solution 3: Monitor-Based Solution (Most Elegant)

Approach: Use a monitor to ensure atomicity

  • Only one philosopher can execute the eating logic at a time
  • Check if both chopsticks are available before picking them up
  • If not available, wait; if available, eat immediately

Advantages:

  • No deadlock
  • No starvation (with proper scheduling)
  • Cleaner, easier to understand

10.6 Key Concepts Demonstrated

This problem teaches:

  • Deadlock: How circular dependencies cause processes to block
  • Starvation: How some processes might never get resources
  • Race Conditions: Multiple processes accessing shared resources
  • Synchronization: Using semaphores and monitors to coordinate
  • Resource Allocation: Managing limited resources among competing processes

10.7 Real-World Applications

  • Database Locks: Multiple transactions needing multiple table locks
  • Network Routing: Packets competing for bandwidth
  • File Systems: Multiple processes accessing files
  • Thread Synchronization: Threads needing multiple locks
